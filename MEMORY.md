# OgorodBot Memory

Этот файл — не дневник, а сжатая рабочая память по `ogorodbot`. Его стоит читать первым перед новыми правками.

## Что это за продукт

- `OgorodBot` — веб-приложение для огорода на `https://ogorod-ai.ru`.
- Главная ценность для холодного пользователя: не “AI вообще”, а ответ на вопрос `что делать сегодня в огороде`.
- Сильные функции продукта:
  - список культур и мест выращивания
  - дневник действий и наблюдений
  - план на 7 дней
  - советы агронома
  - погода и погодные риски
  - бесплатная кнопочная диагностика по симптомам
  - история сезонов и экспорт

## Что уже поняли про упаковку

- Холодный трафик плохо воспринимает широкий месседж “всё для огорода”.
- Главный оффер должен быть узким: `Что делать сегодня` или вторым слоем `План на 7 дней`.
- Диагностика по симптомам хорошо ощущается как реальная ценность, но это второй крючок, не главный первый экран.
- Пользователь явно не любит выдуманные рекламные мокапы: в креативах нужен реальный экран приложения, а не фантазийный UI.

## Текущие продуктовые решения

- Онбординг специально упрощён:
  - спрашивает только город, тип местности, имя по желанию, уровень советов и время уведомлений
  - объекты и культуры теперь добавляются уже после входа во вкладке `Растения`
- Главная вкладка упрощена:
  - убраны `Пульс недели`, `Вечерний фокус` и факты от AI
  - оставлен главный сценарий `Сегодня в огороде`
  - для пустого огорода есть понятное empty state с CTA `Добавить культуры`
- Во вкладке болезней на проде оставлена только кнопочная диагностика.
- Фото-диагностика пока убрана из UI, хотя backend-путь под неё частично готов.

## Подписки и доступы

- Цены в коде:
  - `base`: `150 ₽/мес`
  - `pro`: `300 ₽/мес`
  - `weekly_plan`: `99 ₽` разово
- Что важно помнить:
  - раньше был баг, когда бонусы/реферал могли понизить `pro` до `base`
  - этот баг уже починен на фронте и на сервере
  - legacy-пользователи без структурной `subscription` могут всё ещё существовать, поэтому в коде есть fallback-логика
- Реальные ограничения:
  - культуры: `free 10`, `base 15`, `pro ∞`
  - объекты: `free 1`, `base 3`, `pro ∞`
  - свои сорта: только `pro`
  - недельная погода: `base` и `pro`
  - план на 7 дней: `pro` или отдельная покупка

## Аналитика

- `Яндекс Метрика` интегрирована.
- Счётчик: `108388834`.
- Текущие JS-цели:
  - `view_main`
  - `auth_success`
  - `onboarding_complete`
  - `checkout_opened`
  - `payment_succeeded`
- Важное ограничение:
  - отдельный честкий `page_view/session_start` не был изначально заведен, поэтому “посещения сайта” как историческую метрику раньше показать было нельзя
  - Метрика сейчас частично закрывает этот пробел, но старые данные не восстановить
- Owner analytics существует как скрытый экран:
  - `?view=owner-analytics`
  - доступ завязан на owner identity

## AI и внешние сервисы

- Чат агронома и AI-план недели ходят во внешний `garden-agent`.
- Важная правка от 4 апреля:
  - `VITE_WORKER_API_KEY` больше не светится в клиенте
  - фронт больше не ходит к `garden-agent` напрямую
  - теперь это идёт через `user-data` actions `ask_agronomist` и `load_weekly_plan`
- Фото-диагностика backend-side опирается на `OPENAI_API_KEY`.
- На проде этот secret пока не задан, поэтому живая фото-диагностика не работает.

## Архитектура и код

- Кодовая база уже частично порезана, но не доведена до конца:
  - есть `REFACTORING_PLAN.md`
  - есть `REFACTORING_STEP1_DONE.md` и `REFACTORING_STEP2_DONE.md`
  - `App.tsx` всё ещё остаётся слишком большим и содержит много orchestration-логики
- Реальное состояние рефакторинга:
  - константы, типы, часть hooks и screen-компонентов уже вынесены
  - но финальная цель “тонкий App.tsx” ещё не достигнута
- Недавние важные технические правки:
  - `hashIdentityToUserId` вынесен в единый `shared/identity.ts`
  - тип `ForecastDay` больше не конфликтует с другим одноимённым интерфейсом: в погодном hook он переименован в `WeatherForecastDay`
  - пустые директории `save-push-subscription` и `delete-push-subscription` убраны как мусор

## Где проект хрупкий

- `App.tsx` всё ещё перегружен и остаётся главным местом, где легко снова сделать кашу.
- `OWNER_VK_ID` и `OWNER_EMAILS` всё ещё захардкожены в нескольких местах.
- Тестов нет вообще.
- Дневник до сих пор хранит часть семантики в тексте записи, это хрупко.
- Между фронтом и edge functions ещё есть дублирование типов и контрактов, хотя часть уже приведена в порядок.
- `user-data` — критический backend-узел; если менять auth или action names, легко сломать полпроекта.

## Диагностика по симптомам

- Это уже живая и полезная фича.
- Она должна оставаться бесплатной и понятной.
- Сейчас движок покрывает группы культур:
  - `solanaceae`, `cucurbits`, `brassicas`, `alliums`, `roots`, `leafy`, `berries`, `herbs`, `legumes`, `corn`
- Экран диагностики учитывает удобрения пользователя и должен продолжать это делать.
- Если развивать дальше:
  - сначала улучшать кнопочную диагностику
  - только потом возвращаться к фото-диагностике

## Реклама и креативы

- В папке `ads/` уже много итераций.
- Для следующих рекламных материалов помнить:
  - реальный UI лучше абстрактных мокапов
  - пользователь любит вариант со скрином приложения и понятным сценарием
  - для Директа и VK лучше тестировать узкий оффер, а не перечисление всех фич
- По рекламным метрикам ранний вывод:
  - показы и просмотры есть
  - кликов мало
  - значит, проблема в оффере и первом экране, а не в самой доставке рекламы

## Деплой и эксплуатация

- Фронт деплоится через `gh-pages` командой `npm run deploy`.
- Edge functions деплоятся через `supabase functions deploy ...`.
- Частая особенность:
  - GitHub Pages после деплоя может несколько минут отдавать старый `index.html`
  - поэтому после выката всегда проверять не только локальную сборку, но и реальный HTML/asset hash на проде
- При сборке фронта обязательно учитывать `VITE_YANDEX_METRICA_ID=108388834`, иначе Метрика уйдёт в “спящий” режим.

## Что нельзя забывать при следующих правках

- Не расширять снова первый экран до “всего сразу”.
- Не возвращать прямые клиентские вызовы к `garden-agent`.
- Не дублировать хеш identity ещё раз в новом месте.
- Не ломать fallback-логику подписок для legacy-аккаунтов.
- Перед изменениями в аналитике помнить, что данные уже были загрязнены owner/test-активностью.
- Если снова понадобится фото-диагностика, сначала проверить наличие `OPENAI_API_KEY` в проде.

## Что логично делать дальше, если вернёмся в этот проект

- Вынести owner-конфиг (`OWNER_*`) в env/shared config.
- Добавить хотя бы минимальные тесты на критические контракты:
  - identity hashing
  - подписки / paywall
  - weekly plan / user-data actions
- Дальше продолжать рефакторинг `App.tsx`, но только по живому плану, а не “по ощущениям”.
